## 二、基础权限模型设计

### 2.1 RBAC 权限模型

采用 **RBAC（Role-Based Access Control）** 模型，支持以下核心概念：

#### 2.1.1 用户（User）
系统操作主体，支持多种类型（普通用户、管理员、第三方应用等）

#### 2.1.2 角色（Role）
权限的集合，作为用户和权限之间的桥梁

#### 2.1.3 资源（Resource/Menu）
系统可访问的资源，包括：
- **目录（Directory）**：一级菜单
- **菜单（Menu）**：二级菜单
- **按钮（Button）**：操作按钮
- **接口（API）**：后端接口权限

#### 2.1.4 数据权限（Data Scope）
控制用户可访问的数据范围：
- 全部数据权限
- 本部门及以下数据权限
- 本部门数据权限
- 仅本人数据权限
- 自定义数据权限

### 2.2 扩展功能

#### 2.2.1 多租户支持
- 租户隔离
- 租户级权限配置

#### 2.2.2 动态权限
- URL 动态授权
- 方法级动态权限
- 表达式权限（SpEL）

#### 2.2.3 审计日志
- 登录日志
- 操作日志
- 权限变更日志

## 四、核心功能实现

### 4.1 OAuth2 认证服务器配置

#### 4.1.1 授权服务器配置
- 支持多种授权模式：授权码模式、密码模式、客户端模式、刷新令牌
- JWT 令牌格式，支持自定义 Claims
- Token 持久化到 Redis
- 支持多客户端配置
- 支持多租户

#### 4.1.2 认证流程
```
1. 用户登录 → 2. 认证服务器验证 → 3. 生成 Token 
→ 4. 携带 Token 访问资源 → 5. 网关校验 Token 
→ 6. 资源服务器鉴权 → 7. 返回结果
```

### 4.1.3 PKCE模式流程

```mermaid
sequenceDiagram
    participant User as 用户
    participant Frontend as 前端应用
    participant Gateway as 网关
    participant Auth as 认证服务
    participant System as 系统服务

    User->>Frontend: 1. 访问系统
    Frontend->>Frontend: 2. 生成 code_verifier
    Frontend->>Frontend: 3. 计算 code_challenge
    Frontend->>Auth: 4. 跳转授权请求<br/>(code_challenge + state)
    Note over Auth: 未登录，重定向到登录页
    Auth->>User: 5. 显示登录页面
    User->>Auth: 6. 输入账号密码
    Auth->>Auth: 7. 验证用户身份
    Auth->>Auth: 8. 生成授权码
    Auth->>Frontend: 9. 回调 callback?code+state
    Frontend->>Frontend: 10. 验证 state
    Frontend->>Gateway: 11. POST /oauth2/token<br/>(code + code_verifier)
    Gateway->>Auth: 12. 转发 token 请求
    Auth->>Auth: 13. 验证 code_verifier
    Auth->>Auth: 14. 生成 access_token + refresh_token
    Auth-->>Frontend: 15. 返回 token 响应
    Frontend->>Frontend: 16. 存储 token
    Frontend->>Gateway: 17. 携带 token 访问 API
    Gateway->>Gateway: 18. 验证 token
    Gateway->>System: 19. 转发请求
    System-->>Frontend: 20. 返回业务数据
```

**PKCE模式特点：**
- ✅ 适用于前后端分离架构
- ✅ 无需 client_secret，更安全
- ✅ 防止授权码截获攻击
- ✅ 支持移动端和 SPA 应用

### 4.1.4 Client Credentials 模式流程

```mermaid
sequenceDiagram
    participant Internal as 内部服务
    participant Gateway as 网关
    participant Auth as 认证服务
    participant Resource as 资源服务

    Internal->>Internal: 1. 准备 client_id + client_secret
    Internal->>Gateway: 2. POST /oauth2/token<br/>(grant_type=client_credentials)
    Gateway->>Auth: 3. 转发 token 请求
    Auth->>Auth: 4. 验证客户端凭证
    Auth->>Auth: 5. 生成 access_token
    Auth-->>Internal: 6. 返回 token 响应
    Internal->>Gateway: 7. 携带 token 调用 API<br/>Authorization: Bearer {token}
    Gateway->>Gateway: 8. 验证 token 有效性
    Gateway->>Resource: 9. 转发请求 + 用户信息
    Resource->>Resource: 10. 执行业务逻辑
    Resource-->>Internal: 11. 返回响应
```

**Client Credentials 模式特点：**
- ✅ 适用于微服务间调用
- ✅ 无需用户参与
- ✅ 基于客户端身份认证
- ✅ 高性能场景（无用户上下文）
- ⚠️ 不支持用户级权限控制

### 4.1.5 **两种客户端类型的区别**

| 特性              | 公开客户端 (Public Client) | 保密客户端 (Confidential Client) |
| ----------------- | -------------------------- | -------------------------------- |
| **client_secret** | ❌ **没有**                 | ✅ **有**                         |
| **客户端认证**    | ❌ 无法进行                 | ✅ 必须进行                       |
| **典型场景**      | SPA、移动端、桌面应用      | 后端服务、服务器应用             |
| **PKCE 支持**     | ✅ **必须使用**             | ⚠️ 可选（建议也用）               |
| **示例**          | Vue/React 前端             | Spring Boot 后端                 |

### 7.2 权限标识规范
```
格式：{module}:{action}:{resource}
示例：
- system:user:list      - 系统管理 - 用户管理 - 查询
- system:user:create    - 系统管理 - 用户管理 - 新增
- system:user:update    - 系统管理 - 用户管理 - 修改
- system:user:delete    - 系统管理 - 用户管理 - 删除
```

### 7.3 角色命名规范
```
格式：{scope}_{role_name}
示例：
- admin                 - 超级管理员
```

## 十、链路追踪方案设计

### 10.1 技术架构

采用 **Sleuth + Micrometer Tracing** 理念，实现轻量级链路追踪系统。

#### 10.1.1 核心组件

| 组件名称 | 作用 | 执行时机 |
|---------|------|----------|
| **GatewayTraceFilter** | 网关层生成/提取 TraceId | 请求到达网关时 |
| **LogFilter** | 网关日志记录过滤器 | 请求处理前后 |
| **TraceFilter** | Servlet 环境链路追踪过滤器 | 请求到达微服务时 |
| **TraceAspect** | 方法级链路追踪 AOP 切面 | Service/Controller 方法执行 |
| **TraceFeignInterceptor** | Feign 客户端链路追踪拦截器 | 微服务间调用 |
| **TraceContext** | 线程上下文管理（ThreadLocal） | 整个请求生命周期 |

#### 10.1.2 TraceId 传递流程

```
┌──────────┐    TraceId    ┌──────────┐    TraceId    ┌──────────┐
│  Client  │ ─────────────▶ │ Gateway  │ ─────────────▶ │  System  │
│          │                │          │                │ Service  │
└──────────┘                └──────────┘                └──────────┘
                             │                           │
                             │ X-Trace-ID                │ ThreadLocal
                             │ Header                    │ 传递
                             ▼                           ▼
                        ┌──────────┐               ┌──────────┐
                        │ WebFlux  │               │ Servlet  │
                        │ Context  │               │ Context  │
                        └──────────┘               └──────────┘
```

### 10.2 详细实现

#### 10.2.1 网关层（Gateway）

**GatewayTraceFilter** - 最高优先级全局过滤器

```java
@Component
public class GatewayTraceFilter implements GlobalFilter, Ordered {
    @Override
    public int getOrder() {
        return Ordered.HIGHEST_PRECEDENCE + 1; // 确保第一个执行
    }
    
    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        // 1. 从请求头获取或生成新的 TraceId
        String traceId = getOrGenerateTraceId(exchange);
        
        // 2. 添加到请求头（传递给下游服务）
        // 3. 添加到响应头（返回给客户端）
        
        return chain.filter(exchange);
    }
}
```

**LogFilter** - 网关日志记录

```java
@Component
public class LogFilter implements GlobalFilter, Ordered {
    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        String traceId = exchange.getRequest().getHeaders().getFirst("X-Trace-ID");
        long startTime = System.currentTimeMillis();
        
        // 记录请求日志
        log.info("[Trace] [Gateway] [{}] 收到请求：{} {}", 
                traceId, request.getMethod(), request.getPath());
        
        return chain.filter(exchange).then(Mono.fromRunnable(() -> {
            // 记录响应日志
            long costTime = System.currentTimeMillis() - startTime;
            log.info("[Trace] [Gateway] [{}] 响应完成，耗时：{}ms", traceId, costTime);
        }));
    }
}
```

#### 10.2.2 微服务层（System/Auth Service）

**TraceFilter** - Servlet 环境链路追踪过滤器

```java
@Component
public class TraceFilter extends OncePerRequestFilter {
    @Override
    protected void doFilterInternal(HttpServletRequest request, 
                                   HttpServletResponse response, 
                                   FilterChain filterChain) {
        try {
            // 1. 从请求头获取 TraceId（来自网关）
            String traceId = request.getHeader("X-Trace-ID");
            
            // 2. 设置到 ThreadLocal 中
            if (traceId != null && !traceId.isEmpty()) {
                TraceContext.setTraceId(traceId);
            } else {
                traceId = TraceContext.getOrGenerateTraceId();
            }
            
            // 3. 继续执行过滤链
            filterChain.doFilter(request, response);
            
        } finally {
            // 4. 清理 ThreadLocal，防止内存泄漏
            TraceContext.clear();
        }
    }
}
```

**TraceAspect** - 方法级链路追踪 AOP 切面

```java
@Aspect
@Component
@Order(Ordered.HIGHEST_PRECEDENCE)
@Slf4j
public class TraceAspect {
    
    @Pointcut("execution(* com..service..*(..)) || execution(* com..controller..*(..))")
    public void tracePointcut() {}
    
    @Around("tracePointcut()")
    public Object aroundTrace(ProceedingJoinPoint joinPoint) throws Throwable {
        String traceId = TraceContext.getOrGenerateTraceId();
        String className = joinPoint.getTarget().getClass().getSimpleName();
        String methodName = joinPoint.getSignature().getName();
        long startTime = System.currentTimeMillis();
        
        try {
            log.debug("[Trace] [Aspect] [{}] 开始执行 {}.{}", 
                     traceId, className, methodName);
            Object result = joinPoint.proceed();
            long endTime = System.currentTimeMillis();
            log.debug("[Trace] [Aspect] [{}] 执行完成 {}.{}, 耗时 {}ms", 
                     traceId, className, methodName, (endTime - startTime));
            return result;
        } catch (Throwable e) {
            long endTime = System.currentTimeMillis();
            log.error("[Trace] [Aspect] [{}] 执行异常 {}.{}, 耗时 {}ms - {}", 
                     traceId, className, methodName, (endTime - startTime), 
                     e.getMessage(), e);
            throw e;
        }
    }
}
```

**TraceFeignInterceptor** - Feign 客户端链路追踪

```java
public class TraceFeignInterceptor implements RequestInterceptor {
    @Override
    public void apply(RequestTemplate template) {
        // 从当前线程获取 TraceId
        String traceId = TraceContext.getTraceId();
        
        if (traceId != null && !traceId.isEmpty()) {
            // 添加到 Feign 请求头中
            template.header("X-Trace-ID", traceId);
            log.debug("[Trace] [Feign] [{}] 添加 TraceId 到请求：{}", 
                     traceId, template.url());
        } else {
            // 如果没有则生成新的
            traceId = TraceContext.generateTraceId();
            TraceContext.setTraceId(traceId);
            template.header("X-Trace-ID", traceId);
            log.debug("[Trace] [Feign] [{}] 为请求生成新的 TraceId: {}", 
                     traceId, template.url());
        }
    }
}
```

#### 10.2.3 线程上下文管理

**TraceContext** - 链路追踪上下文

```java
@Slf4j
public class TraceContext {
    // 使用 InheritableThreadLocal 支持父子线程传递
    private static final InheritableThreadLocal<String> TRACE_ID_HOLDER = 
        new InheritableThreadLocal<>();
    
    /**
     * 生成新的 TraceId（16 位 UUID）
     */
    public static String generateTraceId() {
        return UUID.randomUUID().toString().replace("-", "").substring(0, 16);
    }
    
    /**
     * 设置 TraceId 到 ThreadLocal
     */
    public static void setTraceId(String traceId) {
        if (StringUtils.hasText(traceId)) {
            TRACE_ID_HOLDER.set(traceId);
        }
    }
    
    /**
     * 获取当前线程的 TraceId
     */
    public static String getTraceId() {
        return TRACE_ID_HOLDER.get();
    }
    
    /**
     * 获取或生成 TraceId
     */
    public static String getOrGenerateTraceId() {
        String traceId = TRACE_ID_HOLDER.get();
        if (!StringUtils.hasText(traceId)) {
            traceId = generateTraceId();
            TRACE_ID_HOLDER.set(traceId);
        }
        return traceId;
    }
    
    /**
     * 清除 ThreadLocal（防止内存泄漏）
     */
    public static void clear() {
        TRACE_ID_HOLDER.remove();
    }
}
```

### 10.5 使用场景

#### 10.5.1 问题排查

**场景：** 用户反馈某个接口响应慢或报错

**步骤：**
1. 从网关日志获取该请求的 TraceId
2. 使用 ELK/Grafana Loki 搜索该 TraceId 的所有日志
3. 查看完整调用链路和各环节耗时
4. 快速定位性能瓶颈或错误位置

**示例日志：**
```
[Trace] [Gateway] [abc123] 收到请求：POST /api/system/users
[Trace] [Gateway] [abc123] 转发到下游：http://system-service:8083
[Trace] [Servlet] [abc123] 从请求头中提取
[Trace] [Security] [abc123] 内部服务认证成功
[Trace] [Aspect] [abc123] 开始执行 SysUserServiceImpl.addUser
[Trace] [Aspect] [abc123] 执行完成 SysUserServiceImpl.addUser, 耗时 1250ms
[Trace] [Gateway] [abc123] 响应完成，耗时：1280ms
```

#### 10.5.2 性能分析

通过分析同一接口的多个 TraceId 日志，可以：
- 统计平均响应时间
- 识别慢查询
- 发现性能瓶颈

#### 10.5.3 微服务调用链分析

```
Gateway → Auth Service → System Service → Redis/DB
   ↓          ↓              ↓              ↓
TraceId 传递贯穿整个调用链，实现全链路可观测性
```

### 10.6 Trace 日志 vs 操作日志

| 维度 | Trace 日志 | 操作日志 |
|------|-----------|---------|
| **目的** | 技术排查、性能分析 | 业务审计、安全合规 |
| **存储** | 日志文件（ELK 收集） | 数据库（sys_oper_log） |
| **内容** | TraceId、方法名、耗时、异常 | 操作人、操作类型、业务数据 |
| **格式** | 结构化文本 | 关系型数据 |
| **查询** | 通过 TraceId 检索 | 多维度 SQL 查询 |
| **生命周期** | 短期（日志轮转） | 长期（审计要求） |
| **示例场景** | "这个请求为什么失败了？" | "张三昨天修改了哪些数据？" |

**两者互补，缺一不可！**

### 10.7 最佳实践

1. **高优先级**：TraceFilter 必须在所有过滤器之前执行
2. **及时清理**：请求结束后必须调用 `TraceContext.clear()`
3. **线程池注意**：线程池复用线程时需手动传递 TraceId
4. **日志级别**：生产环境建议使用 INFO，开发环境使用 DEBUG
5. **性能考虑**：AOP 切面避免记录过多调试日志
